# IAM

<p className="text-sm text-gray-500">Last updated: 2026-03-22 (d097bdf)</p>


## Principles

Syfrah's IAM is designed for small-to-medium teams (5-50 people). It prioritizes simplicity over granularity. Three decisions drive the entire design:

1. **4 built-in roles, no custom roles.** Enough to cover 90% of use cases.
2. **Roles are per-org or per-project.** Not per-environment, not per-resource.
3. **API keys are per-project.** One key, one project, one permission level.

## Roles

| Role | Intent |
|---|---|
| **Owner** | Runs the platform. Manages users, billing, org settings. |
| **Admin** | Manages infrastructure. Creates projects, VPCs, toggles deletion protection. |
| **Developer** | Builds and deploys. Creates VMs, environments, volumes. Day-to-day work. |
| **Viewer** | Reads everything. Logs, metrics, status. Cannot change anything. |

### What each role can do

| Action | Viewer | Developer | Admin | Owner |
|---|---|---|---|---|
| View resources, logs, metrics | yes | yes | yes | yes |
| Create/start/stop/reboot VMs | | yes | yes | yes |
| Create/destroy environments | | yes | yes | yes |
| Create/delete volumes | | yes | yes | yes |
| Manage security group rules | | yes | yes | yes |
| Create/delete projects | | | yes | yes |
| Create/delete VPCs | | | yes | yes |
| Toggle deletion protection | | | yes | yes |
| Manage API keys | | | yes | yes |
| Manage budgets | | | yes | yes |
| Create/disable users | | | | yes |
| Assign/revoke roles | | | | yes |
| Manage billing | | | | yes |

## Role assignment

Roles are assigned at **org level** or **project level**.

- **Org-level role** = applies to all projects in the org (default)
- **Project-level role** = applies to that project only (override)

If a user has both, the higher privilege wins for that project.

```bash
# Assign at org level (applies everywhere)
syfrah iam assign alice@example.com --role admin --org acme

# Assign at project level (applies to this project only)
syfrah iam assign bob@example.com --role developer --project backend-api

# List assignments
syfrah iam list --org acme

# Revoke
syfrah iam revoke bob@example.com --project backend-api
```

### Typical setup for a 10-person team

```
    Organization: Acme Corp
    │
    │  alice@... → owner (org-level)
    │  bob@...   → admin (org-level)
    │
    ├── Project: backend-api
    │   │  carol@... → developer (project-level)
    │   │  dave@...  → developer (project-level)
    │   │  eve@...   → viewer (project-level)
    │   │
    │   ├── production     (deletion_protection: on)
    │   ├── staging
    │   └── feat/auth-v2   (ttl: 48h)
    │
    └── Project: data-pipeline
        │  carol@... → viewer (project-level, different from backend-api)
        │  frank@... → developer (project-level)
        │
        ├── production     (deletion_protection: on)
        └── staging
```

Carol is a developer on `backend-api` but only a viewer on `data-pipeline`. This is handled by two project-level assignments — no complex policy language needed.

## Protecting production

There are no environment-level roles. Instead, environments have a **`deletion_protection`** flag (see [organization-model.md](organization-model.md)).

When `deletion_protection` is enabled:
- Nobody (not even owners) can destroy the environment without first disabling the flag
- Disabling the flag is an explicit action: `syfrah env update production --no-deletion-protection`
- This is a safety net independent of IAM — it prevents accidents, not unauthorized access

For teams that need "developers can't deploy to production": use a project-level API key scoped to a CI pipeline, and configure the pipeline to only deploy to non-protected environments. This is CI/CD configuration, not IAM.

## Users

### Authentication

Users authenticate with **email + password**. Passwords are stored as bcrypt hashes. Sessions use short-lived JWTs (24 hours).

For the self-hosted version, this is sufficient. No external identity provider needed. No OAuth configuration.

### Creating users

Users are created by the owner. There is no self-registration — this is an infrastructure platform, not a social network.

```bash
# First user: created during org setup
syfrah org create acme --admin-email alice@example.com
# Prompts for password. Alice becomes owner.

# Additional users: created by the owner
syfrah user create --email bob@example.com --name "Bob"
# Generates a temporary password, shown once.
# Bob changes it on first login.

# Login
syfrah login --email bob@example.com
# Prompts for password. Stores session token locally.
```

### Future: OAuth / SSO

When needed (SaaS version or enterprise self-hosted), OAuth (GitHub, Google) is added as an alternative login method. The email address links OAuth identities to existing users. The role model does not change.

## API keys

API keys are for automation: CI/CD, Terraform, scripts. They are scoped to **one project** with a **role** (viewer, developer, or admin — never owner).

### Format

```
    syf_key_a1b2c3d4_7Kx9mQ5pR2wL8nT3...
    │       │         │
    │       │         └── 32 random bytes (base58)
    │       └── project short ID (8 hex)
    └── prefix (scannable, identifiable)
```

The prefix follows the existing `syf_sk_` convention for mesh secrets. Security scanners can detect leaked keys by prefix.

### Creating and using keys

```bash
# Create a key
syfrah apikey create --project backend-api --role developer --name ci-deploy
# → syf_key_a1b2c3d4_7Kx9mQ5pR2wL8nT3...
# Shown once. Only the hash is stored.

# Use in API calls
curl -H "Authorization: Bearer syf_key_a1b2c3d4_7Kx9mQ..." \
  http://[fd12:...]:7100/v1/vms

# Use in CI (e.g., GitHub Actions)
env:
  SYFRAH_API_KEY: ${{ secrets.SYFRAH_DEPLOY_KEY }}

# List keys (never shows the secret)
syfrah apikey list --project backend-api
# NAME          ROLE        CREATED      LAST USED
# ci-deploy     developer   2026-03-15   2h ago
# monitoring    viewer      2026-03-01   5m ago

# Rotate (new key, old key invalidated)
syfrah apikey rotate --project backend-api --name ci-deploy
# → syf_key_a1b2c3d4_NEW_KEY_HERE...

# Delete
syfrah apikey delete --project backend-api --name ci-deploy
```

### Key properties

| Property | Value |
|---|---|
| Scoping | One project |
| Permission | viewer, developer, or admin |
| Expiration | Optional (default: none) |
| Storage | SHA-256 hash only (plaintext shown once) |
| Rotation | Explicit, creates new key and invalidates old |

## Authorization flow

Every API request follows the same path:

```
    Request arrives
         │
    1. Extract identity:
       ├── Bearer token (syf_key_...) → hash, lookup API key → project + role
       └── Session JWT → decode → user + role bindings
         │
    2. Resolve the target resource's project
         │
    3. Look up the effective role:
       ├── API key: role is in the key record
       └── User: highest of org-level and project-level role
         │
    4. Check permission table:
       role_has_permission(role, action) → allow or deny
```

No policy evaluation engine. No rule merging. One table lookup.

## What this does NOT cover (v2+)

- Custom roles / fine-grained permissions
- Per-environment role assignments
- Deploy-only tokens (a developer key in a CI pipeline serves the same purpose)
- Approval workflows ("developer requests, admin approves")
- Temporary access elevation / break-glass
- Audit log (who did what, when)

These are real needs for larger teams. They are not v1 needs. The 4-role model covers a 5-person startup through a 50-person team.

## Comparison

| | **AWS** | **GCP** | **DigitalOcean** | **Syfrah** |
|---|---|---|---|---|
| Roles | 1000+ managed policies | 500+ predefined roles | 3 roles | **4 roles** |
| Custom roles | Yes (unlimited) | Yes | No | **No** |
| Scoping | Per-resource, per-service | Org/folder/project | Account-wide only | **Org or project** |
| API keys | IAM users + access keys | Service accounts + keys | Global read/write | **Per-project + role** |
| Complexity | PhD-level | Master's-level | Too simple | **Just right** |
