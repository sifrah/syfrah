## Executive summary

- **Overall UX score: 6.5/10**
- **Critical issues (P0): 4**
- **Major issues (P1): 11**
- **Minor issues (P2): 14**
- **Positive highlights: 8**

The fabric CLI is functional, has good bones (spinners, non-TTY fallback, sanitized output), and the install script is solid. However, there are several critical consistency problems: `status` leaks the mesh secret in plain output, the README Quick Start uses a command form (`syfrah fabric peering --pin 4829`) that triggers interactive watch mode rather than the documented "set PIN and exit" behavior, and multiple commands use raw `println!` instead of the `ui::` helpers, creating inconsistent output between TTY and non-TTY modes. Error messages are generally actionable, but several commands silently succeed when they should warn about edge cases.

### Positive highlights

1. **ui.rs module is well-designed** -- spinners, TTY detection, non-TTY fallback, styled output. This is a strong foundation.
2. **install.sh is excellent** -- checksum verification, spinners, clear error messages, cleanup on exit.
3. **Error messages reference correct commands** -- most errors tell the operator exactly what to do next.
4. **Leave command is robust** -- stops daemon, tears down WireGuard, clears state, handles partial failures.
5. **Update command with rollback** -- automatic daemon stop/restart, binary backup, rollback on failure. Well-engineered.
6. **Peering watch mode** -- join request cards with visual formatting, Ctrl+C handling, PIN display.
7. **Diagnose command** -- comprehensive health checks across state, daemon, and WireGuard layers.
8. **Input sanitization** -- peer names and mesh names run through `sanitize()` before display.

## Multi-node experience

### Setup flow

**Note:** Multi-node testing was not performed on live servers. This assessment is based on code analysis.

The expected flow:
1. Server 1: `syfrah fabric init --name prod --peering` (or `init` + `peering --pin`)
2. Server 2: `syfrah fabric join <server1-ip> --pin <pin>`
3. Verify: `syfrah fabric status` and `syfrah fabric peers` on both.

**Issue:** The README Quick Start flow uses `syfrah fabric peering --pin 4829` which enters interactive blocking mode. The operator on Server 1 will be stuck in the terminal until Server 2 connects. If the operator opens a second terminal, they lose context. The `--peering` flag on `init` is the correct approach but is undocumented.

### Monitoring flow

- `syfrah fabric status` provides comprehensive info but dumps too much (secret, config tuning values) for routine monitoring.
- `syfrah fabric peers` is the right command for at-a-glance health.
- `syfrah fabric events` provides audit trail.
- No `--watch` mode for status (like `watch -n5 syfrah fabric status`).

### Failure handling

- Unreachable peers: daemon marks them after 5-minute timeout. Status shows "unreachable" count.
- Handshake health: status shows peers with recent handshake (&lt;3min).
- Daemon crash: systemd service auto-restarts (if installed). Without systemd, no recovery.

### Recovery flow

- `syfrah fabric start` restarts daemon from saved state.
- `syfrah fabric rotate` for secret compromise (but requires all peers to rejoin).
- `syfrah fabric leave` + `syfrah fabric join` for a single node rejoin.

### Non-TTY behavior

The `ui.rs` module handles non-TTY well:

- `step_fail` prints `FAIL: <msg>` to stderr.
- `confirm` returns `false` in non-TTY (safe default).

**However**, commands that use raw `println!` bypass this entirely. `status`, `peers`, `events` will print the same output regardless of TTY mode, which is fine for tables but means they miss the styling benefits in TTY mode and the structured format in non-TTY mode.

## Raw notes

1. The `sanitize()` function is applied consistently to user-provided names (mesh name, node name, peer name). Good security practice.

2. The daemon double-fork pattern in `main.rs` is complex but correct. The 5-second polling for PID file is reasonable.

3. `setup_logging` has three modes: daemon (file), JSON (structured), default (stderr). The `SYFRAH_LOG_FORMAT=json` env var is undocumented but useful.

4. The `events.rs` timestamp formatter reimplements date math without chrono. It works but is a maintenance risk (leap second edge cases, etc.).

5. `is_syfrah_process()` reads `/proc/<pid>/cmdline` on Linux, uses `ps` on macOS, returns `true` on other platforms. The fallback `true` on non-Linux/macOS means PID recycling protection is absent on those platforms.

6. The `peering watch` function spawns the daemon in-process (`tokio::spawn`) during auto-init rather than using the normal double-fork path. This means the daemon dies when the peering command exits. This is a secondary consequence of the auto-init behavior (P0 issue PEER-1).

7. The `rotate` command clears `state.peers` but does not notify connected peers. They will continue trying to reach this node with the old secret until they timeout. There is no "broadcast leave" mechanism.

8. The `confirm()` function writes the prompt to stderr (not stdout), which means piping stdout still shows the prompt. Good design choice.

9. The update command's TOCTOU handling (checking daemon state after download, not before) is thoughtful and correctly implemented.

10. `service install` writes a unit file with `Type=simple` which is correct for the `--foreground` exec pattern. `RestartSec=5` and `Restart=always` are appropriate.

11. The README links to `https://github.com/sifrah/syfrah` (note: `sifrah` not `syfrah` -- the org name differs from the project name). This could confuse contributors but is not a bug per se.

12. `wg::interface_summary()` failure in `status.rs` is handled gracefully: `Interface: syfrah0 (down)`. Good.

13. No rate limiting on `peering list` polling in watch mode (1-second interval). For a control socket, this is fine, but documenting the polling behavior would be helpful.

14. The `--foreground` / `-f` flag exists on `init`, `join`, and `start` but is not needed for normal operation. It could confuse new users who see it in `--help`. Consider moving to advanced docs only.
