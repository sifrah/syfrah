## Why this document

The other concept docs describe *what* each layer does. This document describes *who owns the truth* for every piece of state, *how* that state is reconciled into reality, and *what phases* each resource goes through during its lifetime.

This is the implementation framework. If the architecture docs are the blueprint, this is the construction manual.

## Source of truth

Every piece of state in Syfrah has exactly one **source of truth**. Some state is authoritative (persisted in Raft, the single correct answer). Some state is derived (computed from authoritative state + local observation). Some state is observed (reported by the node that owns the resource).

### Fabric state (implemented)

| State | Source of truth | Stored in | Notes |
|---|---|---|---|
| Mesh secret | Init node (generated once) | Raft + local `state.json` | Distributed to joining nodes via peering |
| Mesh prefix (`/48`) | Derived from mesh secret | Computed, not stored separately | Deterministic: `SHA-256("mesh-prefix:" \|\| secret)` |
| Node membership (who is in the mesh) | Raft | Raft log | Authoritative list of all nodes |
| Node WireGuard keypair | Each node (generated locally) | Local `state.json` only | Private key never leaves the node |
| Node mesh IPv6 (`/128`) | Derived from prefix + WG pubkey | Computed, not stored separately | Deterministic: `SHA-256(wg_pubkey)` into prefix |
| Node public endpoint | Each node (self-reported) | Raft (via join) + gossip (updates) | Can change if node's IP changes |
| WireGuard peer config | Derived from node membership | Forge reconciles locally | Each node builds its WG config from Raft membership list |
| Peer liveness (up/down) | Gossip (SWIM probing) | Gossip state (local per-node) | Observed, not authoritative. Used for scheduling hints. |

### Compute state (planned)

| State | Source of truth | Stored in | Notes |
|---|---|---|---|

```
    ┌─────────────────────────────────────────────────────────┐
    │                                                         │

    │                                                         │

    │   org/project/env, users, roles, volumes, snapshots     │
    │                                                         │
    ├─────────────────────────────────────────────────────────┤
    │                                                         │
    │   DERIVED (computed from Raft state by each forge)      │
    │   "What the node should look like right now"            │
    │                                                         │
    │   WG peer config, VXLAN bridges, FDB entries,           │
    │   ARP proxy, nftables rules, NAT, DNS zone files,       │
    │   MAC addresses, mesh IPv6 addresses                    │
    │                                                         │
    ├─────────────────────────────────────────────────────────┤
    │                                                         │
    │   OBSERVED (reported by each forge via gossip)           │
    │   "What is actually happening"                          │
    │                                                         │
    │   VM runtime status, node health, available resources,  │
    │   peer liveness                                         │
    │                                                         │
    └─────────────────────────────────────────────────────────┘
```

**The rule**: Raft is truth. Derived state is recomputed from Raft. Observed state is reported upward. If derived state and observed state disagree, the forge reconciles.

## Reconciliation loop (planned)

> **Not yet implemented.** The reconciliation loop described below is the planned design for the forge layer.

Each node runs a **reconciliation loop** in the forge. It continuously compares what Raft says should exist on this node with what actually exists, and acts on the differences.

### Model

```
    ┌────────────────────────────────────┐
    │          Raft (desired state)       │
    │                                    │
    │  "VM web-1 should run on Node B    │
    │   with 2 vCPU, 4GB, in VPC prod,  │
    │   IP 10.0.1.5, SG web-sg"         │
    └────────────────┬───────────────────┘
                     │
                     │ forge reads Raft state
                     │ for this node
                     ▼
    ┌────────────────────────────────────┐
    │      Forge reconciliation loop      │
    │                                    │
    │  1. List desired resources (Raft)  │
    │  2. List actual resources (local)  │
    │  3. Diff: create missing,          │
    │     update drifted, delete stale   │
    │  4. Report actual state (gossip)   │
    │  5. Sleep, repeat                  │
    └────────────────────────────────────┘
                     │
                     ▼
    ┌────────────────────────────────────┐
    │        Local reality (observed)     │
    │                                    │
    │  Firecracker processes, bridges,   │
    │  TAP devices, nftables rules,      │
    │  ZeroFS volumes, DNS zone files    │
    └────────────────────────────────────┘
```

### How it works

Every N seconds (e.g., 10s), the forge on each node:

**Step 1 — Read desired state from Raft**

The forge reads the subset of Raft state relevant to this node:
- VMs scheduled on this node
- VPCs that have VMs on this node
- Subnets, IP allocations, security groups for those VPCs
- Volumes attached to VMs on this node

**Step 2 — Read actual state from local system**

The forge inspects what actually exists:
- Running Firecracker processes (ps / PID files)
- Linux bridges and VXLAN interfaces (`ip link`)
- TAP devices (`ip link`)
- nftables rules (`nft list ruleset`)
- ZeroFS NBD connections
- DNS zone files

**Step 3 — Diff and act**

| Situation | Action |
|---|---|
| VM in Raft but no Firecracker process | Create TAP, attach to bridge, start Firecracker |
| Firecracker running but VM not in Raft | Stop Firecracker, cleanup TAP |
| VPC in Raft but no bridge on this node | Create VXLAN interface + bridge |
| Bridge exists but VPC removed from Raft | Delete bridge + VXLAN interface |
| Security group rules changed in Raft | Recompute and atomically replace nftables rules |
| FDB entries stale (VM moved to another node) | Remove old FDB, add new |
| Volume attached in Raft but not mounted | Connect ZeroFS NBD, attach to Firecracker |
| DNS record in Raft but not in zone file | Regenerate zone file, reload CoreDNS |

**Step 4 — Report actual state**

After reconciliation, the forge gossips the actual state of each resource:
- VM `web-1` is `Running` (or `Error` if Firecracker crashed)
- Node has 24 vCPU free, 48 GB RAM free
- Health: `Up`

**Step 5 — Sleep and repeat**

The loop runs continuously. It is also triggered immediately on Raft log changes (event-driven), not just on a timer.

### Failure handling

When the forge cannot reconcile a resource:

1. **Retry with backoff** — transient failures (disk full, network blip) are retried 3 times with exponential backoff
2. **Mark as `Failed`** — if retries are exhausted, the resource's observed state is set to `Failed` with an error message
3. **Report via gossip** — the control plane sees the failure and can alert or reschedule
4. **Never silently drop** — a failed resource stays in Raft as desired state until explicitly deleted or the control plane decides to reschedule

The reconciliation loop is **idempotent** — running it twice produces the same result. This means crashes during reconciliation are safe: the next loop iteration picks up where the previous one left off.

### What the reconciliation loop does NOT do

- **Does not make scheduling decisions** — that's the control plane's job
- **Does not modify Raft state** — it only reads Raft and writes to gossip
- **Does not coordinate with other nodes** — each forge reconciles independently
- **Does not retry forever** — after N retries, it marks the resource as failed and moves on

## Resource lifecycles (planned)

> **Not yet implemented.** The phase models below are the planned design. Only the peering join phases are partially reflected in the current fabric implementation.

Every resource type follows a phase model. Phases are linear — a resource moves forward through phases, never backward (except `Failed` → retry → previous phase).

### VM phases

```
    Pending → Scheduled → Provisioning → Running → Stopping → Stopped → Deleting → Deleted
                                           │                     │
                                           └── Failed            └── Failed
```

| Phase | Who sets it | What it means |
|---|---|---|
| **Pending** | Raft (on create) | VM definition exists. Waiting for scheduler. |
| **Scheduled** | Raft (scheduler) | Scheduler picked a node. Waiting for forge to act. |
| **Provisioning** | Forge (gossip) | Forge is creating TAP, bridge, volume, starting Firecracker. |
| **Running** | Forge (gossip) | Firecracker process is alive. VM is operational. |
| **Stopping** | Raft (on stop request) | User requested stop. Forge sending shutdown signal. |
| **Stopped** | Forge (gossip) | Firecracker process exited cleanly. Resources still allocated. |
| **Deleting** | Raft (on delete request) | User requested deletion. Forge cleaning up. |
| **Deleted** | Raft (after cleanup confirmed) | All resources released. VM record can be garbage collected. |
| **Failed** | Forge (gossip) | Something went wrong. Error message attached. |

**Failed is not terminal.** The control plane can retry (reschedule to another node) or the operator can investigate and fix. A failed VM stays in Raft until explicitly deleted.

### Volume phases

```
    Pending → Available → Attaching → Attached → Detaching → Available → Deleting → Deleted
                                        │                                    │
                                        └── Failed                           └── Failed
```

| Phase | Who sets it | What it means |
|---|---|---|
| **Pending** | Raft (on create) | Volume definition exists. ZeroFS file not yet created. |
| **Available** | Forge (gossip) | ZeroFS NBD file created on S3. Not attached to any VM. |
| **Attaching** | Raft (on attach request) | Volume being connected to a VM's Firecracker process. |
| **Attached** | Forge (gossip) | NBD device connected to Firecracker. VM can read/write. |
| **Detaching** | Raft (on detach request) | Volume being disconnected. ZeroFS flushing cache. |
| **Deleting** | Raft (on delete request) | Volume being removed from S3. |
| **Deleted** | Raft (after cleanup) | S3 data removed. Record can be garbage collected. |
| **Failed** | Forge (gossip) | S3 unreachable, ZeroFS error, etc. |

A volume can cycle between `Available` and `Attached` multiple times (attach to VM-A, detach, attach to VM-B).

### VPC phases

```
    Pending → Active → Deleting → Deleted
                │
                └── Degraded
```

| Phase | Who sets it | What it means |
|---|---|---|
| **Pending** | Raft (on create) | VPC definition exists. VNI allocated. No bridges yet. |
| **Active** | Derived (when first VM is placed) | At least one node has a bridge for this VPC. |
| **Degraded** | Gossip (when a node with VMs in this VPC is unhealthy) | Some VMs in this VPC may be unreachable. |
| **Deleting** | Raft (on delete request) | All VMs must be deleted first. Bridges being torn down. |
| **Deleted** | Raft (after cleanup) | VNI released. All bridges removed. |

VPCs are simpler because they are mostly configuration — the actual infrastructure (bridges, VXLAN) is derived state created by the forge.

### Node phases

```
    Joining → Active → Draining → Leaving → Left
                │
                └── Unreachable
```

| Phase | Who sets it | What it means |
|---|---|---|
| **Joining** | Raft (on join request accepted) | Node is syncing Raft state and WireGuard config. |
| **Active** | Gossip (healthy heartbeats) | Node is operational. Scheduler can place VMs here. |
| **Unreachable** | Gossip (failed health checks) | Node stopped responding. VMs may still be running. |
| **Draining** | Raft (operator-initiated) | No new VMs scheduled. Existing VMs being migrated. |
| **Leaving** | Raft (on leave request) | Node is tearing down WireGuard and deregistering. |
| **Left** | Raft (after cleanup) | Node removed from membership. Record can be garbage collected. |

**Unreachable → Active** is automatic if the node starts responding again. **Unreachable → Draining** can be triggered by the control plane after a timeout (to reschedule HA VMs).

### Environment phases

```
    Active → Deleting → Deleted
```

| Phase | Who sets it | What it means |
|---|---|---|
| **Active** | Raft (on create) | Environment exists. Resources can be created in it. |
| **Deleting** | Raft (on delete, after deletion_protection check) | All resources being cleaned up. |
| **Deleted** | Raft (after all resources deleted) | Environment record removed. |

Environments are simple — they are metadata containers. The complexity is in the resources inside them.

For environments with a **TTL**, the control plane transitions `Active → Deleting` automatically when the TTL expires.

### Peering join phases

```
    Requested → Pending → Accepted → Syncing → Joined
                  │
                  └── Rejected
```

| Phase | Who sets it | What it means |
|---|---|---|
| **Requested** | Joining node (TCP) | Join request sent to target node. |
| **Pending** | Target node (local) | Request waiting for operator approval. |
| **Accepted** | Target node (operator) | Operator approved. Mesh secret being transmitted. |
| **Rejected** | Target node (operator) | Operator denied. Connection closed. |
| **Syncing** | Joining node (local) | Received mesh secret + peers. Configuring WireGuard. |
| **Joined** | Both nodes (gossip) | WireGuard tunnel established. Peer announced to mesh. |

## Reconciliation and phases together

The phases are the **language** the reconciliation loop speaks. The loop's job is to move resources from their current phase toward their desired phase.

```
    Example: creating a VM

    Raft state:
      vm-web-1: { desired: Running, spec: {2 vCPU, 4GB, ...} }

    Forge on Node B sees:
      vm-web-1: phase = Scheduled (just assigned to this node)
      No local Firecracker process exists.

    Reconciliation:
      1. Phase is Scheduled → target is Running
      2. No Firecracker process → need to provision
      3. Create TAP, bridge (if needed), volume
      4. Start Firecracker → set phase to Provisioning (gossip)
      5. Firecracker boots → set phase to Running (gossip)
      6. Done. Next loop iteration: Running == Running, no action.
```

```
    Example: node failure

    Gossip detects: Node B is Unreachable

    Control plane (Raft leader) sees:
      vm-web-1: scheduled on Node B, desired: Running
      Node B: Unreachable for >60s

    Control plane acts:
      1. Mark vm-web-1 phase: Pending (reschedule)
      2. Scheduler picks Node C
      3. Mark vm-web-1 scheduled: Node C
      4. Node C's forge reconciles: sees Scheduled, no local process
      5. Provisions and starts the VM
      6. Phase: Running on Node C
```

```
    Example: security group update

    Tenant updates SG rules via API → Raft commits new rules

    Every node hosting VMs with this SG:
      1. Reconciliation loop detects: nftables rules ≠ Raft SG rules
      2. Recomputes nftables ruleset from Raft
      3. Atomically applies new rules (nft -f)
      4. Existing connections preserved (conntrack)
      5. Done. No phase change — SG rules are not a phased resource.
```
