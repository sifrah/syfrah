# Control Plane

<p className="text-sm text-gray-500">Last updated: 2026-03-22 (d097bdf)</p>


## What is the control plane?

The control plane is the brain of the platform. It manages the global state: which VMs exist, where they run, which IPs are allocated, which VPCs are configured. It receives tenant requests ("create a VM"), makes decisions ("place it on Node B"), and coordinates execution across nodes.

Unlike the [forge](forge.md) (which executes on a single node), the control plane operates at the **cluster level**. It sees all nodes, all resources, all tenants.

## Design: every node is the control plane

Syfrah's control plane is **embedded in every node**. There is no dedicated "controller" machine. Every node in the mesh runs the full control plane stack. From the operator's perspective, all nodes are equal.


Internally, one node is elected **Raft leader** for consensus operations. But the leader election is automatic and invisible — the operator never designates a leader. If the leader goes down, another node takes over within seconds.

## Two-layer architecture

The control plane uses two complementary protocols for different types of state:

### Raft — for state that must be consistent

Raft (via `openraft`) provides **strong consistency** through leader-based consensus. All writes go through the leader, are replicated to a majority of nodes, and then committed.

Raft handles state where **conflicts would be dangerous**:
- IP address allocation (never double-allocate)
- VNI assignment (unique per VPC)
- VM scheduling decisions (one node per VM)
- VPC / subnet / security group configuration
- Organization / project / environment definitions
- Volume definitions and attachments

### Gossip — for state that can be eventually consistent

Gossip (SWIM protocol) provides **fast, decentralized propagation** for state that tolerates a few seconds of staleness.

Gossip handles state where **speed matters more than precision**:
- Node health and liveness (heartbeats)
- Available resources per node (CPU, RAM, disk)
- VM runtime status (running, stopped, error)
- VXLAN FDB entries
- DNS record propagation

```
    ┌──────────────────────────────────────────────────────┐
    │                                                      │
    │   Raft (strongly consistent)                         │
    │   "What should exist"                                │
    │                                                      │
    │   VM definitions, IP allocations, VPC config,        │
    │   scheduling decisions, org/project/env              │
    │                                                      │
    │   Writes go to leader → replicated to majority       │
    │   Reads: any node (local state machine)              │
    │                                                      │
    ├──────────────────────────────────────────────────────┤
    │                                                      │
    │   Gossip (eventually consistent)                     │
    │   "What is actually happening"                       │
    │                                                      │
    │   Node health, VM status, resource availability,     │
    │   FDB entries, DNS records                           │
    │                                                      │
    │   Every node publishes and receives via SWIM         │
    │   Convergence: O(log N) rounds (~2-5 seconds)        │
    │                                                      │
    └──────────────────────────────────────────────────────┘
```

The reconciliation loop on each node compares the two layers:
- Raft says "VM-X should run on this node" (desired state)
- Gossip says "VM-X is currently stopped" (actual state)
- The forge acts: start VM-X, then gossip the new status "running"

## How Raft scales from 1 to N nodes

| Nodes | Raft behavior | Fault tolerance |
|---|---|---|
| **1** | Single-node Raft. Leader of itself. Writes are local. | 0 (no replication) |
| **2** | Both must agree. One is leader, one is follower. | 0 (either failing blocks writes) |
| **3** | Quorum of 2. Standard Raft. | 1 failure tolerated |
| **5** | Quorum of 3. | 2 failures tolerated |
| **7+** | Cap voters at 5-7. Extra nodes are learners (replicate data, don't vote). | 2-3 failures tolerated |

**The 1-node case is critical.** Most operators start with a single dedicated server. With 1 node, Raft is degenerate — the node is automatically leader, writes are committed locally, and there is zero overhead. It behaves exactly like a local database. When the operator adds nodes 2 and 3, Raft begins replicating automatically.

**The 2-node case is known to be fragile.** If either node fails, the other cannot form a quorum and writes stop. This is inherent to consensus (not a Syfrah limitation). Existing workloads keep running; only new operations are blocked. The operator should add a third node for fault tolerance.

## API: any node accepts requests

Every node runs an HTTP API server (axum) on its fabric IPv6 address. Tenants and operators can send requests to **any node**.

```
    Tenant sends: POST /v1/vms
         │
    ┌────┴────────────────────────────────────┐
    │  Node B receives the request             │
    │                                          │
    │  Is this node the Raft leader?           │
    │  ├── Yes → process locally               │
    │  └── No  → forward to the leader         │
    │           (transparent to the tenant)     │
    └──────────────────────────────────────────┘
```

The tenant never needs to know which node is the leader. Request forwarding is automatic and transparent.

## Scheduling

When a VM is created, the scheduler picks a node:

```
    1. Read gossip state: available resources per node
    2. Filter: nodes in the right region/zone
    3. Score: bin-packing (prefer nodes with most free resources)
    4. Submit to Raft: "Schedule VM-X on Node B"
    5. Raft commits → all nodes see the decision
    6. Node B's forge creates the VM
```

The scheduler runs on the Raft leader (since it needs to make atomic decisions). It uses gossip data for node capacity but commits decisions through Raft for consistency.

## IP allocation: why Raft matters

IP allocation is the clearest example of why gossip alone is insufficient.

Without consensus: Node A allocates 10.0.0.5 to VM-1. Concurrently, Node B allocates 10.0.0.5 to VM-2. Both are valid local operations. Conflict.

With Raft: both requests go to the leader. The leader processes them sequentially. VM-1 gets 10.0.0.5, VM-2 gets 10.0.0.6. No conflict possible.

```
    Node A: "allocate IP in 10.0.0.0/24"
         │                                    Raft leader
    Node B: "allocate IP in 10.0.0.0/24"     processes
         │                                    sequentially:
         └──────────────┐
                        ▼
                   1. VM-1 → 10.0.0.5
                   2. VM-2 → 10.0.0.6
                        │
                   Replicated to all nodes
```

The IP pool is a bitmap inside the Raft state machine. Only the leader modifies it. Uniqueness is guaranteed by single-threaded log application.

## Node failure detection

Two layers of detection:

### Gossip (fast, ~15-30 seconds)

SWIM protocol detects failed nodes via direct and indirect probing. When a node stops responding to pings, gossip marks it as suspect, then dead. The scheduler immediately stops placing VMs on dead nodes.

### Raft (authoritative, ~60 seconds)

A reconciler task on the Raft leader monitors gossip. When a node is dead for >60 seconds, the leader commits a `NodeDead` event to the Raft log. The state machine marks all VMs on that node as `Lost`. For VMs with restart-on-failure enabled, the scheduler places them on healthy nodes.

```
    Node C stops responding
         │
    T+15s   Gossip marks Node C as dead
            Scheduler stops sending work to C
         │
    T+60s   Raft leader commits NodeDead(C)
            VMs on C marked as Lost
            HA VMs rescheduled to other nodes
         │
    T+65s   Forge on Node A/B starts the rescheduled VMs
```

Existing VMs on healthy nodes are never affected. Only the failed node's VMs need attention.

## State model

Every piece of control plane state belongs to exactly one layer:

### Raft state (strongly consistent)

| State | Description |
|---|---|
| Organizations | Org definitions, billing |
| Projects | Project membership in orgs |
| Environments | Environment definitions, TTL, deletion protection |
| VM definitions | Desired state: vCPU, memory, image, volumes, network |
| Scheduling decisions | Which VM runs on which node |
| VPC configuration | VNI, CIDR, subnets |
| IP allocations | IP → VM mapping (bitmap per subnet) |
| Security group rules | Per-SG firewall rules |
| Volume definitions | Size, attachments |
| Node inventory | Which nodes are in the cluster, their static config |

### Gossip state (eventually consistent)

| State | Description |
|---|---|
| Node health | Up, degraded, draining, dead |
| Available resources | Free CPU, RAM, disk per node |
| VM runtime status | Running, stopped, error (actual, not desired) |
| FDB entries | MAC-to-node mappings for VXLAN forwarding |
| DNS records | VM hostname → IP for private DNS |

**The rule**: if losing or duplicating a piece of state would violate a user-facing invariant (double IP, orphan VM, unauthorized access), it goes in Raft. If staleness just means slightly degraded performance, it goes in gossip.

## Network partition behavior

During a network partition:

| Partition side | Raft | Gossip | Workloads |
|---|---|---|---|
| **Majority** (has quorum) | Continues operating. Accepts writes. | Propagates among majority nodes. | VMs keep running. New VMs can be created. |
| **Minority** (no quorum) | Read-only. Cannot accept writes. | Propagates among minority nodes. | Existing VMs keep running. No new VMs, no IP allocation, no config changes. |

When the partition heals:
- Raft replays missed log entries to the minority side
- Gossip converges within seconds
- The cluster is fully operational again

This is the safest behavior — the minority side is prevented from making conflicting decisions.

## Technology choices

| Component | Technology | Why |
|---|---|---|
| Consensus | `openraft` (Rust) | Async, tokio-native, supports single-node through multi-node. Used in production by Databend. |
| Gossip | `foca` (Rust) | Transport-agnostic SWIM implementation. Works naturally over WireGuard point-to-point links. |
| State store | `redb` (Rust) | Embedded key-value store, pure Rust, ACID, single-file database. No C dependencies. |
| API server | `axum` (Rust) | HTTP framework for the tenant-facing REST API. Lightweight, tokio-native. |

All pure Rust, all embeddable, no external services. Each node is fully self-contained.

## Comparison with other platforms

| | **Kubernetes** | **Proxmox** | **MicroCloud/LXD** | **Syfrah** |
|---|---|---|---|---|
| Consensus | etcd (external) | Corosync (Totem ring) | dqlite (Raft + SQLite) | openraft (embedded Raft) |
| Topology | Centralized (API server + etcd) | Every node equal | Every node equal | **Every node equal** |
| External dependencies | etcd cluster | Corosync daemon | dqlite library (C) | **None** (pure Rust, embedded) |
| WAN tolerance | Poor (etcd needs low latency) | Poor (&lt;5ms required) | Moderate | **Good** (Raft over WireGuard) |
| Min nodes for HA | 3 (etcd) + N workers | 3 (or 2 + QDevice) | 3 | **3** |
| 1-node mode | Requires special setup | Works | Works | **Works** (degenerate Raft) |

The key differentiator: Syfrah's control plane is **embedded** (no external etcd/corosync/dqlite), runs over **WireGuard** (encrypted, WAN-tolerant), and treats every node as **equal** (no server/client distinction).
