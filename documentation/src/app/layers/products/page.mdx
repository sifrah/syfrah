## What are cloud products?

Cloud products are the services exposed to tenants. They are what tenants interact with — the reason the platform exists.

Products are **not** built into the forge. The forge provides low-level primitives (VMs, volumes, network interfaces). Products are orchestration layers that consume these primitives to deliver higher-level services.

## Products vs. primitives

The platform has two distinct layers:

```
    ┌─────────────────────────────────────────────────────────┐
    │                     Cloud Products                       │
    │                                                          │
    │  Each product is an orchestration layer that:            │
    │    • Decides which forge primitives to create            │
    │    • Provisions software and configuration               │
    │    • Manages the service lifecycle (backups, updates)    │
    │    • Exposes a tenant-facing API                         │
    │                                                          │
    ├─────────────────────────────────────────────────────────┤
    │                     Forge (per node)                      │
    │                                                          │
    │  Primitives:                                             │
    │    • Compute  → create/start/stop/delete VMs             │
    │    • Network  → create/delete interfaces, bridges        │
    │    • Storage  → create/delete volumes, snapshots         │
    │                                                          │
    │  The forge is generic. It does not know what runs        │
    │  inside the VMs it creates.                              │
    └─────────────────────────────────────────────────────────┘
```

The forge manages VMs, volumes, and network interfaces. It doesn't know or care what runs inside the VM. A database, a proxy, a customer's application — they're all just VMs with different provisioning.

Products add the **domain knowledge**: what to install, how to configure it, when to back up, how to scale.

## Core pattern: every product is a VM

This is the central insight: **every cloud product is ultimately one or more VMs with specific provisioning on top**.

```
    Product = forge primitives + product-specific configuration
```

A managed database is a VM with database software configured inside. A load balancer is a VM with a reverse proxy configured inside. A tenant VM is a VM with nothing configured — the tenant controls it.

The pattern is always the same:

```
    Control Plane
         │
         │  1. Call forge: create VM, volume, network interface
         │  2. Provision: install and configure software inside the VM
         │  3. Expose: connect to the tenant's VPC, assign endpoint
         │
         ▼
    ┌──────────────────┐
    │  VM running on   │
    │  a node via the  │
    │  forge           │
    │                  │
    │  Software X      │  ← installed and configured by the product layer
    │  configured and  │     (not by the forge)
    │  managed by the  │
    │  platform        │
    └──────────────────┘
```

This keeps the forge simple. New products don't require forge changes — they're just new orchestration logic on top of the same primitives.

## Product lifecycle

Every product follows the same three-phase lifecycle:

### 1. Provision

The control plane calls the forge to create the infrastructure: VM, volumes, network interfaces. This step is **identical across all products** — it uses the same forge REST API.

### 2. Configure

The control plane provisions the software inside the VM. This is where products **diverge** — each product installs different software and applies different configuration. The forge is not involved in this step.

### 3. Expose

The control plane connects the product to the tenant's network (VPC, subnet) and assigns an endpoint. The tenant can now use the service.

```
    PROVISION              CONFIGURE             EXPOSE
    ─────────              ─────────             ──────

    Forge API              Product logic         Overlay + DNS
    (generic)              (product-specific)    (network layer)

    Create VM              Install software      Attach to VPC
    Create volume          Apply configuration   Assign IP/endpoint
    Attach network         Setup backups         Update routing
    Start VM               Setup monitoring      Notify tenant
```

Day-2 operations (backups, upgrades, scaling, failover) follow the same split: the product layer decides **what** to do, the forge executes the **infrastructure** part.

## Provisioning: how software gets into VMs

Three approaches, from simplest to most sophisticated:

| Approach | How it works | Trade-off |
|---|---|---|
| **Pre-built images** | Dedicated rootfs image with software pre-installed. Forge boots the VM with this image. | Fast boot, predictable. Image management overhead. |
| **Cloud-init** | VM boots with a base image, cloud-init runs a provisioning script on first boot. | Flexible, no custom images. Slower first boot. |
| **In-VM agent** | A lightweight agent inside the VM receives configuration from the control plane. | Continuous management, not just first-boot. More complexity. |

The initial implementation will likely use **pre-built images** for speed, with **cloud-init** for per-instance customization.

## Why this separation matters

The forge/product split has several consequences:

**For product development:** adding a new product means writing orchestration and provisioning logic. No changes to the forge, no changes to the fabric, no changes to existing products.

**For the forge:** it stays small and generic. It doesn't accumulate product-specific logic. Every product team gets the same primitives.

**For tenants:** they interact with product APIs (create a database, create a load balancer), never with the forge directly. The forge is invisible to them — just like EC2's host agent is invisible to AWS customers.

**For operations:** the forge is the single place to manage compute resources on a node. Products don't bypass it — every VM, volume, and interface goes through the same API.
