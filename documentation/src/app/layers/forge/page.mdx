## What is the forge?

The **forge** is the local control service that runs on every node in the fabric. It exposes a REST API bound exclusively to the fabric interface (`syfrah0`), making it reachable only from within the mesh — never from the public internet.

The forge is the bridge between the control plane and the local compute stack. It receives instructions ("create a VM", "attach a network", "take a snapshot") and executes them on the local machine. It also reports the node's state back to the control plane: available resources, running workloads, health status.

```
    ┌─────────────────────────────────────────────────────────┐
    │                     Cloud Products                       │
    │         (VMs, Load Balancers, Managed PostgreSQL)        │
    ├─────────────────────────────────────────────────────────┤
    │                    Forge API                        │
    │          REST/HTTP on fabric (syfrah0 only)              │

    ├─────────────────────────────────────────────────────────┤
    │                  Internal Compute Stack                   │

    │                    │   VXLAN)     │   snapshots)         │
```

## Why REST/HTTP?

The forge uses plain HTTP (not HTTPS) over the fabric. This is a deliberate choice:

| Concern | How it's handled |
|---|---|
| **Encryption** | WireGuard encrypts all traffic on syfrah0 (ChaCha20-Poly1305) |
| **Authentication** | Only nodes in the mesh can reach the API (fabric isolation) |
| **Debuggability** | `curl http://[fd12:...]:7100/status` works from any mesh node |
| **Simplicity** | No certificate management, no TLS termination, no PKI |
| **Performance** | No TLS handshake overhead on every request |

HTTPS would add a second layer of encryption inside the already-encrypted WireGuard tunnel — complexity for no security gain. The fabric provides the same guarantees that TLS would: confidentiality, integrity, and peer authentication.

This is the same approach used by Kubernetes kubelet (HTTP on internal network), AWS Nitro hypervisor agents, and most cloud provider host agents.

## Network binding

The forge listens **only** on the node's fabric IPv6 address, not on `0.0.0.0` or `::`:

```

    │  Public interface (eth0):                │
    │    51.210.x.x  ← NOT listening here      │
    │                                          │
    │  Fabric interface (syfrah0):             │
    │    [fd12:3456:7800:a1b2:...]:7100        │
    │    ▲                                     │
    │    └─ Forge API bound here          │
    │       Only reachable from the mesh       │
```

This means:
- A port scan from the internet will never find the forge
- Only authenticated mesh members can reach it
- No firewall rules needed — the binding itself is the access control

## Architecture

### Layers

Each node runs three layers, from bottom to top:

```
    ┌────────────────────────────────────────────────┐
    │                 Forge API                   │
    │                                                  │
    │  GET  /status         → node health + resources  │
    │  GET  /info           → node identity + config   │
    │  POST /compute/vms    → create a VM              │
    │  ...                  → (future endpoints)       │
    ├────────────────────────────────────────────────┤
    │              Internal Compute Stack               │
    │                                                  │
    │  ┌────────────┐ ┌───────────┐ ┌──────────────┐  │
    │  │ Compute    │ │ Network   │ │ Storage      │  │
    │  │            │ │           │ │              │  │
    │  │ Firecracker│ │ bridges   │ │ local disks  │  │
    │  │ microVMs   │ │ tap devs  │ │ volumes      │  │
    │  │ lifecycle  │ │ VXLAN     │ │ snapshots    │  │
    │  └────────────┘ └───────────┘ └──────────────┘  │
    ├────────────────────────────────────────────────┤
    │                   Fabric                         │
    │            WireGuard mesh (syfrah0)              │
    └────────────────────────────────────────────────┘
```

**Forge API** — The REST layer. Receives requests from the control plane (or from operators via curl). Translates API calls into operations on the internal compute stack.

**Internal Compute Stack** — The actual machinery: Firecracker for VMs, Linux bridges and tap devices for networking, local disks for storage. This layer is never exposed directly — it's only accessible through the forge.

**Fabric** — The transport. Provides encrypted, authenticated connectivity between nodes.

### Relationship with the control plane

The forge is a **worker**, not a **coordinator**. It doesn't make scheduling decisions. It doesn't know about other nodes' workloads. It simply executes what it's told and reports its own state.

```
    ┌──────────────────────────────┐
    │         Control Plane         │
    │    (future, centralized)      │
    │                              │
    │   "Place VM X on Node B"     │
    │   "Attach VPC subnet to      │
    │    Node B's bridge"          │
    └──────────┬───────────────────┘
               │
               │  REST/HTTP over fabric
               │
    ┌──────────┴──────────┐  ┌─────────────────────┐
    │   Node A Agent      │  │   Node B Agent       │
    │  fd12:...:A:7100    │  │  fd12:...:B:7100     │
    │                     │  │                      │
    │  "I have 32 vCPU,   │  │  "Creating VM X...   │
    │   128 GB RAM,       │  │   Attaching network  │
    │   2 VMs running"    │  │   ...done."          │
    └─────────────────────┘  └──────────────────────┘
```

The control plane calls individual forges to:
1. **Query state** — what resources are available, what's running
2. **Execute actions** — create/start/stop/delete VMs, configure networking, manage storage
3. **Collect metrics** — CPU usage, memory pressure, disk I/O, network traffic

The forge is **stateless in terms of intent** — it doesn't persist "desired state" or reconcile. It executes commands and reports reality. The control plane is responsible for desired state, reconciliation, and retry logic.

## API design

### Principles

1. **Resource-oriented** — endpoints map to resources (`/compute/vms`, `/network/bridges`, `/storage/volumes`)
2. **JSON in, JSON out** — standard REST with JSON bodies
3. **Synchronous for fast ops, async for slow ops** — creating a VM returns immediately with a task ID; the caller polls or watches for completion
4. **Idempotent where possible** — retrying a create with the same ID doesn't create duplicates
5. **Node-scoped** — the API only knows about local resources on this specific node

## Internal compute stack

The forge delegates to the internal compute stack. This stack is not exposed via API — it's a set of local subsystems managed by the agent:

### Compute subsystem (Firecracker)

- Manages Firecracker microVM processes
- Handles VM lifecycle: create, start, stop, delete
- Manages kernel images and root filesystem images
- Configures vCPU count, memory, and boot parameters

### Network subsystem

- Creates and manages Linux bridges (one per VPC subnet on this node)
- Creates tap devices for VM network interfaces
- Configures MAC and IP addresses
- Handles overlay encapsulation (VXLAN or similar) for cross-node VM traffic

### Storage subsystem

- Manages local block devices and disk images
- Creates and mounts volumes for VMs
- Handles snapshots and backups (to object storage)

```
    │  │                  │  │               │  │

    │              syfrah0 (fabric)             │
    │              fd12:3456:...:xxxx           │
    └──────────────────────────────────────────┘
```

**syfrah daemon** — Manages the fabric: WireGuard mesh, peering, peer health, control socket for CLI. Already implemented.

**forge** — Manages compute workloads: VM lifecycle, local networking, storage. Listens on a different port on the same fabric address. New component.

Both processes share the fabric interface but are independent. The daemon handles infrastructure; the agent handles workloads.

## Security considerations

### Access control

Today: any node in the mesh can call any other node's agent API. This is acceptable for a single-operator mesh (one person/team controls all nodes).

Future: the control plane will be the only caller of forge APIs. Access control will be enforced by:
1. **Token-based auth** — the control plane includes a bearer token derived from the mesh secret
2. **Role separation** — only the control plane role can call mutation endpoints; nodes can only call `/status` on each other

### Blast radius

The forge has root-level access to the local machine (it manages WireGuard, Firecracker, bridges, volumes). A compromised agent means a compromised node. This is inherent to the hypervisor agent model — same as AWS Nitro, GCP Borglet, or any host agent.

Mitigations:
- Agent bound to fabric only (no internet exposure)
- Fabric membership requires operator approval
- Future: signed requests from control plane, audit logging

### Input validation

The forge must validate all inputs strictly:
- VM resource requests must not exceed node capacity
- Network configurations must reference valid VPCs/subnets
- Volume sizes must respect available disk space
- IDs must be well-formed (no path traversal, no injection)

The agent is a **trust boundary** — it sits between the control plane (which may relay tenant requests) and the local operating system. Every input is untrusted until validated.

## Operator experience

From any node in the mesh, an operator can query other nodes directly:

```bash
# Check node health
curl http://[fd12:3456:7800:a1b2:c3d4:e5f6:7890:abcd]:7100/status

# List VMs on a remote node
curl http://[fd12:3456:7800:a1b2:c3d4:e5f6:7890:abcd]:7100/compute/vms

# Create a VM on a specific node (for testing/debugging)
curl -X POST http://[fd12:3456:7800:...]:7100/compute/vms \
  -H "Content-Type: application/json" \
  -d '{"id":"test-vm","vcpu":1,"memory_mb":512,"rootfs_image":"ubuntu-24.04"}'
```

This makes the forge immediately useful even before the control plane exists. An operator can manually orchestrate VMs across nodes using curl — the same way early Kubernetes clusters could be managed with direct kubelet API calls.
