## Choice: redb

Syfrah uses [redb](https://github.com/cberner/redb) as the embedded key-value store for all persistent state across layers.

### Why redb

| Criterion | redb | Alternatives considered |
|---|---|---|
| **Pure Rust** | Yes, zero C dependencies | sled (abandoned), jammdb (abandoned) |
| **ACID** | Yes, crash-safe by default | sled (partial ACID, known corruption bugs) |
| **Single file** | Yes, one `.redb` file per layer | fjall (directory-based LSM), heed (directory-based LMDB) |
| **Concurrency** | MVCC (unlimited readers + 1 writer, non-blocking) | rusqlite (WAL mode, similar but C dependency) |
| **Maintenance** | v3.1.1, actively maintained (2026) | sled (stalled since 2024), jammdb (abandoned 2023) |
| **File format** | Stable since 1.0, migration paths guaranteed | sled (unstable, warns of breaking changes) |
| **Footprint** | Minimal memory, small binary | rusqlite (bundles full SQLite C library) |

### Alternatives rejected

- **sled**: 8,953 stars but unmaintained, known data corruption bugs, never reached 1.0, self-described as "beta". README says "if reliability is your primary constraint, use SQLite."
- **fjall**: Good LSM-tree engine but directory-based storage, overkill for small datasets (hundreds of keys per layer).
- **heed**: LMDB wrapper, not pure Rust (C dependency via FFI). Excellent performance but violates the no-C-dependency principle.
- **rusqlite**: Most battle-tested option (SQLite), but C dependency. Overkill for key-value access patterns. Would be the fallback if we ever need SQL queries.
- **jammdb**: Pure Rust BoltDB clone, nice design, but abandoned since November 2023.
- **persy**: Pure Rust, ACID, single file, but tiny community (42K downloads/quarter vs redb's 1.3M), no public benchmarks, infrequent updates.

## Architecture: one file per layer

Each layer gets its own redb file. No shared database across layers.

```
    ~/.syfrah/
    ├── fabric.redb          ← fabric layer state
    ├── compute.redb         ← compute layer state (future)
    ├── overlay.redb         ← overlay layer state (future)
    ├── storage.redb         ← storage layer state (future)
    ├── controlplane.redb    ← control plane state (future)
    └── org.redb             ← organization layer state (future)
```

### Why one file per layer

**Isolation.** A layer cannot corrupt another layer's data. A bug in overlay's write path cannot destroy fabric's peer records.

**Independent lifecycle.** `syfrah fabric leave` deletes `fabric.redb` without touching other layers. Each layer manages its own data lifecycle.

**No write contention.** redb allows one writer at a time per database file. With separate files, the fabric layer writing peers does not block the overlay layer writing FDB entries. Each layer has its own write lock.

**Follows the architecture.** The repo is organized by layer (one crate, one directory, one README). The state follows the same boundary: one file per layer.

### Why NOT a shared file

A single shared redb file would allow cross-layer atomic transactions (e.g., create VM + allocate IP in one transaction). However:

- **Cross-layer coordination is the control plane's job.** The Raft log provides atomicity for multi-step operations. The database doesn't need to.
- **Global write lock.** One writer at a time across ALL layers means fabric writes block overlay writes. Unacceptable as the system grows.
- **Dangerous cleanup.** `syfrah fabric leave` with a shared file requires surgically deleting only fabric tables, not the whole file. With separate files, it's just `rm fabric.redb`.

## Table design

Each layer defines its own tables. Tables are strongly typed at compile time via redb's `TableDefinition`.

### Fabric layer tables

```rust
use redb::TableDefinition;

// Mesh configuration (single-row key-value pairs)
const CONFIG: TableDefinition<&str, &[u8]> = TableDefinition::new("config");
// Keys: "mesh_name", "mesh_secret", "wg_private_key", "wg_public_key",
//        "mesh_ipv6", "mesh_prefix", "wg_listen_port", "node_name",
//        "public_endpoint", "peering_port"

// Peer records (keyed by WG public key)
const PEERS: TableDefinition<&str, &[u8]> = TableDefinition::new("peers");
// Key: wg_public_key (base64 string)
// Value: serde-serialized PeerRecord

// Metrics (counters)
const METRICS: TableDefinition<&str, u64> = TableDefinition::new("metrics");
// Keys: "peers_discovered", "wg_reconciliations",
//        "peers_marked_unreachable", "daemon_started_at"
```

### Future layer tables (examples)

```rust
// Overlay layer
const VPCS: TableDefinition<&str, &[u8]> = TableDefinition::new("vpcs");
```

## Usage pattern

### Opening

Each layer opens its database file at startup:

```rust
use redb::Database;
use std::sync::Arc;

let db = Arc::new(Database::create("~/.syfrah/fabric.redb")?);
// Share `db` across tokio tasks via Arc<Database>
```

The `Database` handle is `Send + Sync`. Wrap in `Arc` and clone into async tasks.

### Writing (transactions)

All writes go through transactions. This solves the store race condition (the current JSON load+modify+save is not atomic):

```rust
let write_txn = db.begin_write()?;
{
    let mut table = write_txn.open_table(PEERS)?;
    table.insert(wg_public_key, &serde_json::to_vec(&peer_record)?)?;
}
write_txn.commit()?;
// Committed to disk. Crash-safe.
```

Multiple writes in the same transaction are atomic — either all succeed or none do.

### Reading

Reads use read transactions (non-blocking, concurrent with writes):

```rust
let read_txn = db.begin_read()?;
let table = read_txn.open_table(PEERS)?;

// Get one peer
let peer_bytes = table.get(wg_public_key)?.map(|v| v.value().to_vec());

// Iterate all peers
for entry in table.iter()? {
    let (key, value) = entry?;
    let peer: PeerRecord = serde_json::from_slice(value.value())?;
    // ...
}
```

### Deleting a layer's state

```rust
// Equivalent of current store::clear()
std::fs::remove_file("~/.syfrah/fabric.redb")?;
```

Simple file deletion. No need to drop tables or vacuum.

## What this replaces

The current fabric store (`layers/fabric/src/store.rs`) uses:
- `~/.syfrah/state.json` — atomic write via temp file + rename
- `serde_json::to_string_pretty` for serialization
- Full file rewrite on every save (every 30 seconds + every peer change)

The new store uses:
- `~/.syfrah/fabric.redb` — ACID transactions, no full rewrite
- `serde_json::to_vec` for values (same serialization, stored as bytes in redb)
- Incremental updates (insert/update one peer, not rewrite all state)

### What this fixes

| Problem | JSON file | redb |
|---|---|---|
| Concurrent writes (store race condition) | Last write wins, data lost | ACID transaction, serialized writes |
| Crash during write | Possible corruption (mitigated by atomic rename) | Crash-safe by design (copy-on-write B-tree) |
| Full state rewrite every 30s | O(N) write for N peers | O(1) for single-key update |
| State file corruption | Manual recovery needed | Automatic recovery (redb handles) |

## Conventions

| Convention | Rule |
|---|---|
| File location | `~/.syfrah/&#123;layer&#125;.redb` |
| One file per layer | No shared databases |
| Table definitions | `const` in the layer's store module |
| Value serialization | `serde_json::to_vec` / `serde_json::from_slice` |
| Database handle | `Arc<Database>`, shared across async tasks |
| Write transactions | Short-lived, commit immediately |
| Read transactions | Non-blocking, can overlap with writes |
