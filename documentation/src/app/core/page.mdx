# Core

<p className="text-sm text-gray-500">Last updated: 2026-03-25 (8fed48c)</p>


## Purpose

The `syfrah-core` crate is the foundation that every other layer depends on. It contains shared types, validation logic, cryptographic primitives, and pure functions. It performs no I/O, no networking, and no async — it is strictly a library of deterministic building blocks.

## Responsibilities

- Mesh secret generation, parsing, and key derivation (`MeshSecret`).
- IPv6 ULA mesh prefix generation and deterministic node address derivation.
- Node identity types (`NodeIdentity`).
- Peer record types and encrypted peer exchange (`PeerRecord`, `JoinRequest`, `JoinResponse`).
- Peering protocol message types (`PeeringMessage`).
- Input validation for names, WireGuard keys, endpoints, and peer records.
- AES-256-GCM encryption and decryption of peer records.

## Non-goals

- **No I/O.** Core never reads from disk, opens sockets, or calls external processes.
- **No async.** No tokio runtime, no futures. Everything is synchronous and pure.
- **No CLI commands.** Core has no user-facing commands; it is consumed as a library.
- **No catch-all.** Types used by only one layer belong in that layer, not here. Only types referenced by two or more layers live in core.

## Public concepts

| Concept | Description |
|---|---|
| **Mesh secret** | A 32-byte cryptographic secret (`syf_sk_...`) that bootstraps the entire mesh. All keys and addresses derive from it. |
| **ULA addressing** | Each mesh gets a random IPv6 ULA `/48` prefix. Each node gets a deterministic `/128` address derived from the prefix and its WireGuard public key. |
| **Node identity** | A node's human-readable name paired with its WireGuard public key. |
| **Peer record** | The full description of a mesh peer: name, key, endpoint, IPv6 address, status, region, and zone. |
| **Peering messages** | Protocol messages exchanged during node join and peer announcement. |

## Failure modes

Core functions are pure and deterministic. The only failures are:

| Failure | Cause | Handling |
|---|---|---|
| `SecretError` | Malformed mesh secret string (bad prefix, encoding, or length) | Returned as `Result`; caller decides how to report |
| `MeshError::Validation` | Invalid input to a validation function (name too long, bad key format, etc.) | Returned as `Result`; caller rejects the input |
| `MeshError::EncryptionFailed` | AES-256-GCM encryption failure (should not happen with valid key) | Returned as `Result` |
| `MeshError::DecryptionFailed` | Wrong key or corrupted ciphertext | Returned as `Result`; caller treats as untrusted data |

There are no runtime panics, no I/O errors, and no timeouts.

## Tests

Run core tests:

```
cargo test -p syfrah-core
```

All tests are unit tests that run without root, without network, and without any external dependencies. Coverage includes:

- `secret.rs` — generation, roundtrip serialization, distinct derivations, error cases (bad prefix, bad encoding, wrong length).
- `addressing.rs` — ULA prefix format, prefix preservation, determinism, uniqueness across keys.
- `identity.rs` — serde roundtrip.
- `mesh.rs` — validation functions, encryption/decryption roundtrip, error cases.
